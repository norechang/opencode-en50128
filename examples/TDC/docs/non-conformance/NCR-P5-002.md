## Non-Conformance Description

**Title**: VER Failed Independent Verification of Coverage Waiver

**EN 50128 Clause(s)**: §6.2.4.14 (VER Independence), §7.5.4.10(c) (VER Coverage Verification), §5.1.2.10e (Independent Verification at SIL 3-4)

**Description**:

VER accepted TST's "hardware dependency" justification for 40% coverage gap in Item 20 (Software Component Test Report) **without performing independent verification** as required by EN 50128 §7.5.4.10(c).

**Specific VER Failures**:

1. **Did NOT check stub inventory**: VER did not verify whether stubs existed for the claimed hardware dependencies. The `tests/stubs/` directory contains stubs for ALL 5 modules TST claimed were "not testable":
   - `hal_stub.c` (for `hal_services.c`)
   - `tci_deps_stub.c` (for `tci_init.c`, `tci_tx.c`)
   - `fmg_deps_stub.c` (for `fmg_init.c`)
   - `obd_dsm_stub.c` (for `dsm_init.c`)

2. **Did NOT verify per-line untestability**: VER did not identify specific uncovered lines and verify that each was genuinely not testable with existing test infrastructure.

3. **Accepted unverified assertion**: VER's statement in Item 19 (DOC-SOURCECODEVER-2026-001 v1.0) — *"83% of uncovered code is justified by hardware-dependent paths"* — was an **unverified assertion** based solely on TST's claim, not on independent VER analysis.

4. **Did NOT apply EN 50128 Hard Rule**: EN 50128 §7.5.4.10(c) requires VER to "verify that the coverage achieved is adequate" and "verify that untestable code is justified". VER did not apply the necessary rigor:
   - No cross-reference between TST's coverage waiver and stub inventory
   - No verification that stubs were inadequate to test the claimed paths
   - No demand for per-line justification or Table A.19 static analysis

**Consequence**: VER's approval of Item 20's coverage waiver undermined the independence and integrity of Phase 5 verification. The waiver was conditionally approved by VMGR based on VER's flawed analysis, which propagated the error to Track B approval.

## Root Cause

**Primary**: VER did not apply the independence principle required by EN 50128 §6.2.4.14. VER accepted TST's self-assessment without performing an independent check of the evidence.

**Contributing Factors**:
- VER may have lacked awareness of the `tests/stubs/` directory structure (tooling/process gap)
- Time pressure in Phase 5 Track B may have led VER to accept TST's rationale at face value
- No VER checklist item for "Verify stub inventory before accepting hardware-dependency justification"

## Impact Assessment

- **Independence Integrity**: HIGH — VER's role as an independent verifier (SIL 3 mandatory per §5.1.2.10e) was compromised. VER effectively rubber-stamped TST's conclusion without verification.
- **V&V Process Credibility**: HIGH — VMGR's conditional approval was based on VER's flawed analysis. This undermines confidence in the entire Phase 5 V&V outcome.
- **Lifecycle Impact**: Phase 5→6 gate check CANNOT proceed until VER re-performs source code verification against corrected Item 20.

## Required Corrective Action

**VER SHALL**:

1. **Re-perform source code verification** (Item 19) after TST submits reworked Item 20 (see NCR-P5-001).

2. **Apply EN 50128 Hard Rule** for coverage verification (§7.5.4.10(c)):
   - **Step 1**: Read the reworked Item 20 coverage report and identify all uncovered lines
   - **Step 2**: Cross-check each uncovered line against the `tests/stubs/` directory to verify whether a stub exists for the claimed hardware dependency
   - **Step 3**: For each uncovered line, demand one of the following from TST:
     - Additional test case that exercises the line (if stub is adequate)
     - Proof that existing stub cannot simulate the path (technical justification)
     - Table A.19 static analysis evidence proving the line is correct (data flow, control flow, or formal proof)
   - **Step 4**: Accept coverage waiver ONLY if all three verification steps are satisfied

3. **Update VER Checklist** (add to SVP §3.2 VER Procedures):
   - Checklist item: *"Before accepting any hardware-dependency justification, verify that no stub exists in `tests/stubs/` for the claimed dependency. If a stub exists, demand additional test cases or proof that the stub cannot test the path."*

4. **Document VER methodology** in reworked Item 19:
   - Section "Coverage Verification Methodology" explaining the 4-step process above
   - Table listing each uncovered line, stub availability, and VER verification result

5. **Re-submit Item 19** to QUA for format-gate review (iteration 2).

**Target Date**: Phase 5 Rework completion (1 week from NCR date).
