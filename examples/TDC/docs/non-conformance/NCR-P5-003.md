## Non-Conformance Description

**Title**: Compound Condition Coverage Not Measured

**EN 50128 Clause(s)**: EN 50128:2011 Table A.21 Requirement 2 (Test Coverage for Code), SVP §3.3 (Coverage Targets)

**Description**:

The Software Component Test Report (Item 20, DOC-COMPTESTRPT-2026-001 v1.0) **does not include compound condition coverage results** for any module.

**EN 50128 Requirement**:
- **Table A.21 Requirement 2** (Test Coverage for Code) specifies for SIL 3:
  - Statement coverage: **R** (Recommended)
  - Branch coverage: **HR** (Highly Recommended)
  - **Compound condition (or equivalent) + branch coverage: HR** (Highly Recommended)
  - Data flow coverage: **R** (Recommended)
  - **For Table A.21 Req. 2 at SIL 3, one of the following combinations is expected**: Branch + Compound Condition (HR), OR Branch + Data Flow (R+R), OR Statement + Data Flow (R+R)

**SVP Commitment**:
- SVP DOC-SVP-2026-001 §3.3 defines **100% compound condition coverage** as a **target** for safety-critical modules (SKN, SPM, OBD, DSM, FMG, TCI, HAL) at SIL 3

**Item 20 Omission**:
- Section 2.3 "Coverage Results" reports:
  - Statement coverage: 60.1%
  - Branch coverage: 55.1%
  - **Compound condition coverage: NOT REPORTED**
- Item 20 states: *"Compound condition coverage is not claimed at this phase."*
- **No rationale provided** for the omission
- **No formal deviation approval** documented

**Analysis**:
1. EN 50128 Table A.21 Req. 2 classifies branch + compound condition coverage as **HR (Highly Recommended)** for SIL 3 component testing
2. SVP §3.3 commits to 100% compound condition coverage for safety-critical modules
3. Item 20 provides neither measurement results nor a justified deviation
4. This constitutes non-compliance with both EN 50128 Table A.21 and the project's own SVP commitment

## Root Cause

**Primary**: TST did not configure the coverage measurement toolchain (gcovr) to collect compound condition coverage data. Only statement and branch coverage were measured.

**Contributing Factors**:
- gcovr requires `--branch-coverage` flag for branch coverage and `--decisions` flag for compound condition (decision) coverage
- TST may have been unaware of the EN 50128 Table A.21 Req. 2 requirement for compound condition (or equivalent) coverage at SIL 3
- SVP §3.3 lists compound condition as a target but does not mark it as mandatory, which may have led TST to deprioritize it

## Impact Assessment

- **EN 50128 Compliance**: HIGH — Table A.21 Req. 2 requires branch + compound condition (HR) or an equivalent combination. Omitting compound condition measurement without justification is a compliance gap.
- **Safety Impact**: MEDIUM — Compound conditions (e.g., `if (A && B)`) may contain logic errors that branch coverage alone does not detect. Untested compound condition paths may harbor latent defects in safety-critical modules.
- **Lifecycle Impact**: Phase 5→6 gate check CANNOT proceed until compound condition coverage is measured and reported, or a formal deviation is approved by VMGR.

## Required Corrective Action

**TST SHALL**:

1. **Measure compound condition coverage** for all safety-critical modules (SKN, SPM, OBD, DSM, FMG, TCI, HAL):
   - Configure gcovr with `--decisions` flag to collect compound condition (decision) coverage
   - Re-execute all 106 test cases with compound condition coverage enabled
   - Generate compound condition coverage report (HTML/JSON)

2. **Report compound condition coverage** in the reworked Item 20:
   - Add a "Compound Condition Coverage" subsection to §2.3 "Coverage Results"
   - Provide per-module compound condition coverage percentages
   - Provide overall compound condition coverage percentage
   - Reference machine-readable evidence (gcovr JSON/HTML)

3. **Justify any coverage gaps** per SVP §3.3 target (100% for safety-critical modules):
   - If compound condition coverage is <100%, provide per-condition technical rationale
   - If any compound condition is genuinely not testable, prove correctness via Table A.19 static analysis (control flow analysis, Boolean algebra simplification, or formal proof)

4. **Update Item 20** and re-submit to QUA for format-gate review (iteration 2).

**Alternative** (if toolchain does not support compound condition coverage):
- TST may propose an **equivalent combination** per Table A.21 Req. 2:
  - Branch + Data Flow coverage (R+R), OR
  - Statement + Data Flow coverage (R+R)
- If alternative is proposed, TST must:
  - Document the rationale in Item 20
  - Obtain VMGR approval for the deviation from SVP §3.3

**Target Date**: Phase 5 Rework completion (1 week from NCR date).
