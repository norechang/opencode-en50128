## Non-Conformance Description

**Title**: TST Coverage Waiver Invalid — Test Inadequacy Masquerading as Hardware Dependency

**EN 50128 Clause(s)**: §7.4.4.8 (Component Testing), §7.5.4.10(c) (VER Coverage Verification), Table A.21 Req. 2 (SIL 3 coverage requirements), SVP §3.3 (100% statement/branch/compound condition coverage target for safety-critical modules)

**Description**:

TST produced a Component Test Report (Item 20, DOC-COMPTESTRPT-2026-001) with coverage results significantly below SVP targets:
- **Statement Coverage**: 60.1% (580/965 lines covered)
- **Branch Coverage**: 55.1% (236/428 branches covered)
- **Compound Condition Coverage**: NOT MEASURED (see NCR-P5-003)

TST justified the 40% coverage gap as "hardware-dependent paths not testable at component level" in the following 5 modules:
1. `hal_services.c`
2. `tci_init.c`
3. `tci_tx.c`
4. `fmg_init.c`
5. `dsm_init.c`

**HOWEVER**: A QUA post-approval audit discovered that the project **already has comprehensive test stubs** for ALL claimed hardware dependencies:

| Module Under Test | Stub File | Location |
|-------------------|-----------|----------|
| `hal_services.c` | `hal_stub.c` | `tests/stubs/hal_stub.c` |
| `tci_init.c`, `tci_tx.c` | `tci_deps_stub.c` | `tests/stubs/tci_deps_stub.c` |
| `fmg_init.c` | `fmg_deps_stub.c` | `tests/stubs/fmg_deps_stub.c` |
| `dsm_init.c`, `dsm_fsm.c` | `obd_dsm_stub.c` | `tests/stubs/obd_dsm_stub.c` |

**Evidence that stubs are functional**:
- The test setUp() functions in `test_hal.c`, `test_tci.c`, `test_fmg.c`, and `test_dsm.c` successfully call hardware initialization functions (`HAL_Init()`, `TCI_Init()`, `FMG_Init()`, `OBD_DSM_Init()`) via these stubs
- The stubs compile cleanly and link with the Unity test framework
- 106/106 test cases PASS, proving the stub infrastructure is operational

**Conclusion**: The 40% coverage gap is NOT caused by hardware dependencies — it is caused by **test inadequacy** (insufficient test cases written). TST mischaracterized the root cause to justify low coverage, which undermines the integrity of the Phase 5 verification.

## Root Cause

**Primary**: TST did not write sufficient test cases to exercise all testable execution paths in the 6 identified modules. Test infrastructure (stubs) exists and is functional, but test cases were incomplete.

**Contributing Factor**: Time pressure in Phase 5 may have led TST to accept incomplete test suites and rationalize the gap with an invalid hardware-dependency justification.

## Impact Assessment

- **Safety Impact**: MEDIUM — Safety-critical modules (OBD, DSM, FMG, TCI) have untested code paths that may contain latent defects. SIL 3 requires high confidence in component correctness before integration.
- **Verification Integrity**: HIGH — VER accepted TST's coverage waiver based on an invalid justification (see NCR-P5-002). This undermines the independence and rigor of Phase 5 V&V.
- **Lifecycle Impact**: Phase 5→6 gate check CANNOT proceed until coverage targets are met or genuinely untestable paths are proven correct via Table A.19 static analysis.

## Required Corrective Action

**TST SHALL**:

1. **Write additional test cases** for all 6 under-tested modules to achieve SVP §3.3 coverage targets:
   - `hal_services.c`
   - `tci_init.c`
   - `tci_tx.c`
   - `fmg_init.c`
   - `dsm_init.c`
   - `dsm_fsm.c` (also identified as under-tested in coverage report)

2. **Target**: 100% statement coverage, 100% branch coverage per SVP §3.3. If any path is genuinely not testable with existing stubs, it must be:
   - Explicitly identified (file, function, line number)
   - Justified with technical rationale (why stub cannot simulate this path)
   - Proven correct via Table A.19 static analysis (data flow analysis, control flow analysis, or formal proof)

3. **Update Item 20** (Software Component Test Report) with:
   - New test case descriptions (test ID, module, coverage objective)
   - Re-executed coverage results (statement, branch, compound condition)
   - Revised coverage waiver requests (if any genuinely untestable paths remain)
   - Machine-readable evidence (JUnit XML, gcovr JSON/HTML)

4. **Re-submit Item 20** to QUA for format-gate review (iteration 2).

**Target Date**: Phase 5 Rework completion (1 week from NCR date).
